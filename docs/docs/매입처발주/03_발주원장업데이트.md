## 2. 처리 대상

구글시트 `캐리어_발주원장`에서 완료여부 = **발주중** 인 행만 업데이트 대상으로 모은다.
완료여부 = **발주완료** 인 행은 따로 모아서, 거기 적힌 오더번호를 "이미 처리된 오더번호"
집합으로 별도 보관한다 (5번 항목에서 중복 처리 방지용으로 사용).

## 3. 캐리어 회신파일 읽기 — `판매오더생성양식` 시트 컬럼 위치

캐리어 회신파일은 openpyxl로 열어 `판매오더생성양식` 시트를 읽는다. 열 위치는
고정되어 있다고 가정한다 (1-index 기준):

| 열 | 의미 |
|---|---|
| A | 오더번호 |
| Q | 받는고객명 |
| R | 받는전화 |
| S | 받는핸드폰 |
| T | 받는우편번호 |
| U | 받는주소1 |
| W | 설치고객명 |
| X | 설치전화 |
| Y | 설치핸드폰 |
| Z | 설치우편번호 |
| AA | 설치주소1 |
| AE | 아이템(모델코드) |

```text
전화번호 = (받는전화 or 받는핸드폰) 우선,  없으면 (설치전화 or 설치핸드폰)
주소     = 받는주소1 우선,  없으면 설치주소1
```
전화번호는 하이픈/공백을 제거해 정규화하고, 주소는 공백·쉼표·괄호를 제거한 뒤
**앞 20자만** 비교에 사용한다 (`clean_phone`, `clean_addr` 함수).

오더번호 또는 모델코드가 없는 행은 건너뛴다.

## 4. 매칭 로직

01_매입처발주.md에 적힌 "1순위 핸드폰, 2순위 일반전화, 3순위 주소" 설명은 매칭 *방법을
로그에 표시할 때 쓰는 라벨*에 더 가깝다. 실제 코드 동작은 이렇다:

```text
구글시트의 발주중 목록을 순서대로 훑다가,
"모델코드 일치" AND
("실구매자휴대폰 일치" OR "실구매자전화번호 일치" OR "실구매자주소 앞20자 일치")
조건을 처음 만족하는 행을 찾으면 그 행으로 확정한다.
(이미 오더번호가 채워진 행은 후보에서 제외)
```
세 조건 중 어느 것으로 맞았는지에 따라 콘솔에 `매칭방법`(핸드폰/일반전화/주소)을
표시하지만, 우선순위를 매겨서 더 좋은 매칭을 고르는 구조는 아니다 — 발주중 목록
순서상 먼저 나오는 행이 채택된다.

매칭에 성공하면 구글시트에 **오더번호**, **완료여부(→발주완료)** 두 셀을 함께
업데이트한다. 매칭에 실패하면 ⚠️ 경고만 출력하고 구글시트는 비워둔 채로 둔다
(→ 현황판 오더번호 미입력 탭에서 수동 입력 필요).

## 5. 이미처리 판별

회신 데이터의 오더번호가 "이미 발주완료 처리된 오더번호" 집합에 있으면, 매칭을
시도하지 않고 곧바로 "ℹ️ 이미 처리됨"으로 건너뛴다. 같은 회신 파일을 다시 실행해도
안전한 이유가 이것이다.

## 6. ⚠️ 회신파일이 여러 개일 때 — 미처리 리포트가 부정확할 수 있음

코드를 읽으면서 발견한 부분이다. 매칭에 사용하는 "남은 발주중 목록"
(`남은발주중`)이 **회신파일 1개를 처리할 때마다 매번 새로 초기화**된다:

```python
for fpath in 캐리어_파일:           # 회신파일이 여러 개면 이 루프가 여러 번 돈다
    ...
    남은발주중 = list(발주중_rows)   # ← 매번 전체 목록으로 리셋됨
    ...
    for 회신 in 회신_데이터:
        ...                        # 매칭되면 이 안에서 남은발주중에서 제거
    if updates:
        ws.batch_update(updates)    # 구글시트에는 정상적으로 즉시 반영됨

# 루프가 끝난 뒤, 마지막 파일 처리 시점의 남은발주중만 남아있음
미처리 = [(i, r) for i, r in 남은발주중 if ...]   # ← 이전 파일들의 매칭 결과가 반영 안 됨
```

**실제 영향:** 구글시트 업데이트 자체는 파일 단위로 즉시 반영되므로 정확하다.
문제는 모든 파일 처리가 끝난 뒤 콘솔에 찍는 "회신 없는 발주중 건" 요약 메시지뿐이다
— 회신파일이 2개 이상이고 그중 앞쪽 파일에서 이미 매칭된 행이, 뒤쪽 파일 처리 후의
`남은발주중`에는 다시 포함되어 있어서 미처리로 잘못 표시될 수 있다.

**대응:** 회신파일을 한 번에 1개씩만 원본 폴더에 넣고 실행하면 이 문제는 발생하지
않는다 — 현재 운영 방식(캐리어 회신파일을 받을 때마다 그때그때 처리)과 맞아떨어져서
실무에서는 거의 부딪힐 일이 없을 것으로 보이지만, 혹시 한 번에 여러 파일을 몰아서
처리할 경우 콘솔 요약은 참고만 하고 구글시트 현황판으로 다시 확인하는 게 안전하다.

## 8. 처리 대상

구글시트 `윈세이_발주원장`에서 **사방넷주문번호가 있고 송장번호가 비어있는** 행만
업데이트 대상으로 모은다. 완료여부 값은 보지 않는다 — 발주완료 이후 어느 시점이든
송장번호만 비어있으면 대상이 된다.

## 9. 매칭 로직

```text
발주총괄(사방넷 재다운로드) 파일에서
매입처 = 'kyungginambupoonglim' (윈세이) 인 행만 필터
        ↓
{사방넷주문번호: 택배송장번호} 맵 생성 (둘 다 값이 있는 행만)
        ↓
구글시트의 미입력 대상 행과 사방넷주문번호로 정확히 일치하는 것만 매칭
        ↓
일치하면 송장번호 칸만 채움
```
모델코드나 전화번호 같은 보조 매칭 기준은 쓰지 않는다 — 사방넷주문번호 자체가
고유 식별자라 정확히 일치하는 것만 본다.

**완료여부는 이 프로그램이 건드리지 않는다.** 01_매입처발주.md·02_매입처발주서생성.md
에 적힌 윈세이 완료여부(발주대기→발주완료)는 이미 2단계(발주서생성.py)에서 끝나며,
3단계는 송장번호만 채우는 역할이다.
