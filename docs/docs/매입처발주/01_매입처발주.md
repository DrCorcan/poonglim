## 3. 캐리어 처리 로직

### 3-1. 대상 주문상태
```python
대상상태_캐리어 = ['출고대기', '신규주문', '주문확인']
```
세 상태 모두 일단 가져온다. 그 안에서 완료여부를 다르게 부여한다.

### 3-2. 완료여부 분기 (핵심)

```
신규주문 / 주문확인
   → 완료여부 = 해피콜대기  (발주 전 대기, 재고 영향 없음)

출고대기 + 선매입재고 있음 (캐리어_선매입재고상세에 대기중 행 존재)
   → 완료여부 = 기존재고
   → 선매입재고상세 해당 행: 대기중 → 사용중
   → 선매입재고현황 수량 -1

출고대기 + 선매입재고 없음
   → 완료여부 = 신규  (2단계에서 발주서 생성 대상)
```

선매입재고 매칭 기준: `설치점코드` + `모델코드 앞 10자` 일치하는 첫 번째 **대기중** 행.

### 3-3. 해피콜대기 → 신규 자동 전환

같은 사방넷주문번호가 이미 구글시트에 있는데(중복), 그 건의 기존 완료여부가
`해피콜대기`이고 이번에 읽은 발주총괄에서는 주문상태가 `출고대기`로 바뀌어 있다면:

```
완료여부: 해피콜대기 → 신규  로 자동 업데이트
```

이렇게 해서 "해피콜 끝나고 사방넷에서 출고대기로 바꾸면, 다음 매입처발주.py 실행 시
자동으로 발주 대상이 된다"는 흐름이 만들어진다.

### 3-4. 중복 처리
사방넷주문번호가 이미 구글시트 `캐리어_발주원장`에 있으면 **추가하지 않는다.**
(단, 3-3의 해피콜대기→신규 전환은 예외적으로 상태만 업데이트한다.)

### 7-1. 전화번호 2개 컬럼이 필요한 이유
발주총괄의 `인수자 핸드폰번호`와 `인수자 일반번호`는 둘 다 캐리어에 전달되는데,
**캐리어가 둘 중 어느 쪽으로 회신할지 예측할 수 없다.** 한 번호로만 저장하면
3단계(오더번호업데이트.py)에서 매칭이 실패하는 사례가 실제로 발생했다
(예: 와이프가 캐리어에 일반번호로 전달했는데 원장엔 핸드폰번호만 있던 경우).
그래서 두 번호 모두 저장해서, 3단계에서 두 번호 모두로 매칭을 시도한다.

| 컬럼 | 소스 |
|---|---|
| 실구매자휴대폰 | 발주총괄 `인수자 핸드폰번호` |
| 실구매자전화번호 | 발주총괄 `인수자 일반번호` |

### 7-2. 주문자명 컬럼이 필요한 이유
발주총괄의 `인수자` 컬럼은 `"수취인명 주문자명 채널코드"` 형태로 들어온다.
대부분은 수취인=주문자라 동일하지만, 가끔 다른 경우가 있다
(예: `"김상윤 문종우 SF"` → 수취인은 김상윤, 주문자는 문종우).

| 컬럼 | 추출 방법 |
|---|---|
| 실구매자명 | 인수자 첫 번째 단어 (수취인명) |
| 주문자명 | 인수자 두 번째 단어 (주문자명) |

### 7-3. 설치점구분 판단
`물류처` 컬럼 값이 설치점 목록(`INSTALL_POINTS`, 36개 등록됨)에 있으면 `배송구분=설치점`,
없으면 `배송구분=직송`으로 분류한다.
