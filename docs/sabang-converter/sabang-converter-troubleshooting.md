## 1. 문서 목적

이 문서는 `sabang.py` 실행 결과에서 발견되는 오류를 **거래처별 사례가 아니라 증상 패턴 중심**으로 정리한다.

원칙:

```text
구체적인 거래처명, 상품코드, 공급단가, 택배비 등 실제 변경 이력은 changelog.md에만 기록한다.
이 문서는 증상, 원인 후보, 확인 방법, 조치 방향만 기록한다.
실제 현재값은 거래처_매핑테이블.xlsx와 거래처_단가테이블.xlsx를 확인한다.
```

구체 사례가 필요하면 `sabang-converter-changelog.md`의 해당 날짜 변경 이력을 참고한다.

## 2. 기본 판단 원칙

```text
프로그램 실행 성공 = 사방넷 업로드 가능이 아니다.
```

실행 로그에서 아래를 반드시 확인한다.

```text
거래처 식별 여부
변환 건수
모델명 출력값
공급단가
택배비
주문번호 출처
합계/총계 행 포함 여부
세트/트레이 경고 여부
시트 선택 경고 여부
```

## 3. 증상 패턴: 거래처 식별 실패

### 증상

```text
❌ 파일명 → 거래처 식별 불가
```

### 원인 후보

```text
거래처_매핑테이블.xlsx에 거래처 행이 없음
파일명키워드가 실제 파일명과 맞지 않음
파일명에 불필요한 띄어쓰기, 약칭, 특수문자가 있음
```

### 확인 방법

```text
1. 파일명에 포함된 거래처명을 확인한다.
2. 거래처_매핑테이블.xlsx의 파일명키워드와 비교한다.
3. 거래처명이 단가테이블의 거래처명과도 동일한지 확인한다.
```

### 조치 방향

```text
매핑테이블에 거래처 신규 행을 추가하거나 파일명키워드를 보정한다.
신규 거래처라면 단가테이블에도 같은 거래처명을 추가한다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-19 변경 이력을 참고한다.

## 4. 증상 패턴: 단가테이블 거래처 누락

### 증상

```text
거래처 식별은 됐지만 공급단가가 비정상이다.
모델명이 원본 상품명 일부로 남는다.
상품코드 매칭이 되지 않는다.
```

### 원인 후보

```text
거래처_매핑테이블에는 거래처가 있으나 거래처_단가테이블에는 거래처가 없음
거래처명이 두 테이블에서 다르게 입력됨
```

### 확인 방법

```text
1. 매핑테이블의 거래처명과 단가테이블의 거래처명이 정확히 같은지 확인한다.
2. 단가테이블에 해당 거래처의 상품 행이 최소 1개 이상 있는지 확인한다.
3. 실행 로그에서 공급단가가 공란이거나 예상과 다른지 확인한다.
```

### 조치 방향

```text
거래처_단가테이블.xlsx에 같은 거래처명으로 상품 행을 추가한다.
실제 값은 단가테이블에만 입력하고 문서에는 중복 기재하지 않는다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-19 변경 이력을 참고한다.

## 5. 증상 패턴: 상품코드 미매칭

### 증상

```text
모델명이 사방넷모델명으로 변환되지 않는다.
모델명 칸에 원본 상품명 일부가 출력된다.
새 상품인데 기존 상품 단가가 나온다.
```

### 원인 후보

```text
단가테이블의 상품코드 키워드가 원본 상품명/모델명/옵션 안에 없음
원본 발주서에서 모델명 또는 옵션 컬럼이 매핑되지 않음
상품코드 키워드가 너무 넓거나 너무 좁음
```

### 확인 방법

```text
1. 원본 발주서의 상품명, 모델명, 옵션 값을 확인한다.
2. 단가테이블의 상품코드가 해당 텍스트 안에 포함되는지 확인한다.
3. 사방넷모델명이 기대값으로 출력됐는지 확인한다.
4. 공급단가가 해당 거래처의 다른 상품 단가로 잘못 빠졌는지 확인한다.
```

### 조치 방향

```text
단가테이블에 적절한 상품코드 키워드를 추가한다.
필요하면 매핑테이블에서 H_모델명 또는 O_옵션 컬럼 연결을 보완한다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-17~2026-06-19 변경 이력을 참고한다.

## 6. 증상 패턴: 기본단가 오사용

### 증상

```text
실행은 성공했지만 공급단가가 이상하다.
새 상품인데 거래처의 기존 상품 단가가 나온다.
모델명이 원본 그대로 또는 원본 일부로 남아 있다.
```

### 원인 후보

```text
현재 sabang.py 구조상 상품코드 매칭 실패 시 거래처의 첫 번째 단가가 기본단가처럼 사용될 수 있음
단가테이블에 상품코드가 없는데도 실행이 중단되지 않음
```

### 확인 방법

```text
1. 해당 거래처 단가테이블의 첫 번째 상품 단가와 출력 단가를 비교한다.
2. 모델명이 사방넷모델명으로 변환됐는지 확인한다.
3. 신규 상품인지 확인한다.
```

### 조치 방향

```text
해당 거래처의 단가테이블에 상품코드를 추가한다.
향후 프로그램 개선 시 미매칭이면 기본단가를 쓰지 않고 오류로 표시하는 안전장치를 검토한다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-19 변경 이력을 참고한다.

## 7. 증상 패턴: 합계/총계 행 포함

### 증상

```text
수취인에 합계, 총계, 소계 같은 값이 들어간다.
수량이나 모델명이 공란인 행이 생성된다.
실제 주문보다 변환 건수가 많다.
```

### 원인 후보

```text
원본 발주서의 합계 행이 데이터 범위 안에 포함됨
매핑테이블에 행 필터 조건이 없음
수취인 컬럼만으로 빈 행을 제외하기 어려움
```

### 확인 방법

```text
1. 실행 로그의 변환 건수와 실제 주문 건수를 비교한다.
2. 검증 체크리스트의 수취인, 모델명, 수량 공란 여부를 본다.
3. 원본 발주서에서 실제 주문행과 합계행을 구분하는 컬럼을 찾는다.
```

### 조치 방향

```text
매핑테이블의 업체필터컬럼과 업체필터값을 사용한다.
같은 거래처를 여러 매핑행으로 나누지 않고, 1개 행 안에서 필터를 적용한다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-19 변경 이력을 참고한다.

## 8. 증상 패턴: 시트 선택 경고

### 증상

```text
[주의] 당일 시트 자동 선택 실패 → 첫 번째 시트 사용
```

### 원인 후보

```text
시트명이 오늘 날짜 패턴과 맞지 않음
실제 발주 시트명이 Sheet1, 당일발주 등 고정명임
여러 시트 중 첫 번째 시트를 자동 사용함
```

### 확인 방법

```text
1. 실제 발주 데이터가 선택된 시트에 있는지 확인한다.
2. 다른 시트에 더 최신 데이터가 있는지 확인한다.
3. 변환 건수가 원본 주문 건수와 맞는지 확인한다.
```

### 조치 방향

```text
선택된 시트가 실제 발주 시트이면 치명 오류로 보지 않는다.
반복적으로 문제되는 거래처는 시트 선택 규칙 보완을 검토한다.
```

구체 사례는 `sabang-converter-changelog.md`의 2026-06-19 변경 이력을 참고한다.

## 9. 증상 패턴: 주문번호가 자동생성됨

### 증상

```text
검증 체크리스트 주문번호 앞에 ★ 표시가 없다.
prefix + 날짜 + 순번 형태로 주문번호가 생성된다.
```

### 원인 후보

```text
원본 발주서에 주문번호 컬럼이 없음
매핑테이블의 P_주문번호가 비어 있음
원본 주문번호 컬럼은 있으나 해당 행 값이 비어 있음
```

### 확인 방법

```text
1. 원본 발주서에 주문번호 컬럼이 있는지 확인한다.
2. 매핑테이블의 P_주문번호가 해당 컬럼으로 연결되어 있는지 확인한다.
3. 실행 로그에서 ★ 발주서 주문번호 사용 건수를 확인한다.
```

### 조치 방향

```text
거래처 발주서에 주문번호가 있으면 P_주문번호를 연결한다.
원본 주문번호 값이 있으면 원본을 우선 사용하고, 비어 있으면 자동생성한다.
```

## 10. 증상 패턴: 세트/트레이 확인 경고

### 증상

```text
PFW-901 감지: 세트 여부 확인 필요
세트 상품 감지: 사방넷 입력 시 2행 분리 필요
```

### 원인 후보

```text
세트, 트레이, + 모델, 경고 대상 모델이 원본 상품명 또는 사방넷모델명에 포함됨
단품과 세트가 같은 모델코드를 공유할 수 있음
```

### 확인 방법

```text
1. 원본 상품명에 세트, 트레이, 포함, + 같은 표현이 있는지 확인한다.
2. 단품 발주인지 세트 발주인지 확인한다.
3. 사방넷 입력본에서 행 분리가 필요한지 확인한다.
```

### 조치 방향

```text
단품이면 그대로 진행한다.
세트이면 단가테이블의 사방넷모델명 또는 세트 분리 로직을 확인한다.
```

## 11. 증상 패턴: 택배비 이상

### 증상

```text
수량이 많은데 택배비가 1회만 붙는다.
제주 주소인데 추가 택배비가 반영되지 않는다.
합포장인데 추가 택배비가 예상과 다르다.
```

### 원인 후보

```text
단가테이블의 기본 택배비 또는 제주추가택배비가 다름
상품별 수량 택배비 예외 규칙이 없음
현재 프로그램의 수량별 택배비 보정 대상이 제한적임
```

### 확인 방법

```text
1. 단가테이블의 택배비와 제주추가택배비를 확인한다.
2. 주소에 제주 여부가 포함되어 있는지 확인한다.
3. 수량별 별도 택배비 기준이 있는 상품인지 확인한다.
```

### 조치 방향

```text
단가테이블을 수정하거나 상품별 택배비 예외 규칙을 코드에 추가한다.
업무 기준이 확정되지 않은 경우 업로드 전에 수동 확인한다.
```
