## 1. 윈세이 — 마감 구조

### 1-1. 원장 컬럼 (윈세이_발주원장, 총 24개 — 헤더명 기준, 순서 고정 아님)

```
접수일 / 주문상태 / 점포명 / 고객명 / 연락처 / 우편번호 / 주소 / 모델코드 /
수량 / 공급가_VAT제외 / 합계_VAT포함 / 물류메시지 / 사방넷주문번호 / 수집일자 /
완료여부 / 송장번호 /
─── 아래 8개가 이번 작업으로 신규 추가됨 ───
송장입력일 / 반품여부 / 반품표시일 / 윈세이환입상태 /
윈세이환입신청일 / 윈세이환입완료일 / 환입차감금액_VAT포함 / 최종처리일시
```

### 1-2. 마감 기준

- **양수(마감매입)**: 송장번호가 처음 입력된 시점 = **송장입력일**. 평소 업무(송장번호 입력)는 그대로, 다만 그 순간 송장입력일이 같이 기록됨.
- **음수(환입차감)**: 윈세이환입상태=완료 + 윈세이환입완료일 기준.
- 원장에는 실제 음수 행을 추가하지 않음. 화면(suppliers.html)에서만 가상 음수 행으로 표시.

### 1-3. 취소 — `cancelWinseiOrder(saBangNo)`

```
조건: 송장번호 공란 + 완료여부가 발주대기/발주완료 + 이미 주문취소가 아님
처리: 완료여부 → 주문취소, 최종처리일시 기록. 매입 영향 없음.
화면: 송장번호 있으면 취소 불가 — "반품 절차를 사용하세요" 안내.
```

### 1-4. 반품 — 3단계, 완료여부는 절대 안 바뀜 (발주완료 유지)

| 단계 | 함수 | 조건 | 처리 | 금액 영향 |
|---|---|---|---|---|
| ① 반품됨 표시 | `markWinseiReturn` | 송장번호 있음, 완료여부≠주문취소, 반품여부 공란 | 반품여부=반품됨, 반품표시일=오늘 | 없음 |
| ② 환입 신청 | `requestWinseiReturn` | 반품여부=반품됨, 윈세이환입상태 공란 | 윈세이환입상태=신청, 신청일=오늘 | 없음 |
| ③ 환입 완료 | `completeWinseiReturn(saBangNo, returnDate, deductionAmount)` | 반품여부=반품됨, 윈세이환입상태=신청 | 환입상태=완료, 완료일/차감금액 저장 | **차감 발생** |

차감금액 검증: `0보다 커야 함`, `원 주문금액(합계_VAT포함)을 초과할 수 없음`.

**중요 — 변심 반품은 이 흐름 대상 아님**: 변심이면 재포장해서 직접 재판매하므로 윈세이와의 거래는 그대로 유지됨. 이미 풍림_회수AS관리(hoisu-as.html)에서 반품접수→회수완료 흐름으로 별도 추적되고 있어, 이 화면에 중복 입력하지 않음. (사방넷 반품/회수/교환 자동 연동은 그 처리 흐름 자체가 아직 명확히 정의되지 않아 의도적으로 보류 — 향후 과제.)

### 1-5. 화면 — `getWinseiOrders()` 데이터로 클라이언트에서 계산 (서버에 별도 집계 함수 없음)

```
이번달 마감건수/마감매입 = 송장입력일이 이번달인 행
이번달 환입차감          = 윈세이환입상태=완료 & 윈세이환입완료일이 이번달인 행의 환입차감금액 합
이번달 순매입            = 마감매입 - 환입차감
```

윈세이 서브뷰는 내부에 탭 2개를 가짐: **📊 발주현황** / **❌ 취소·반품**.
- 발주현황 탭: 카드 6개 + 마감카드 4개(마감건수/마감매입 클릭 시 마감내역 토글, 기본 접힘) + 표
- 취소·반품 탭: 주문번호 직접 조회 또는, "반품됨"/"환입신청" 카드 클릭 시 **해당 상태 전체 목록**이 먼저 뜨고 거기서 "처리" 버튼으로 주문을 선택 (카드가 특정 주문을 임의로 선택하지 않음)

## 2. 스타리온 — 마감 구조

### 2-1. 원장 컬럼 (스타리온_발주원장, 총 21개)

```
접수일 / 주문상태 / 점포명 / 고객명 / 연락처 / 우편번호 / 주소 / 모델코드 /
수량 / 공급가_VAT제외 / 합계_VAT포함 / 물류메시지 / 사방넷주문번호 / 수집일자 /
완료여부 /
─── 아래 6개가 이번 작업으로 신규 추가됨 ───
설치완료여부 / 설치완료일 / 취소상태 / 취소확정일 / 취소차감금액_VAT포함 / 최종처리일시
```

### 2-2. 핵심 차이 — 캐리어와 다름, 윈세이와도 다름

```
캐리어: 취소돼도 오더번호 있으면 우리 재고로 편입됨
윈세이: 재고관리 안 함, 반품(불량만)이 있음
스타리온: 재고관리 안 함, 반품 개념 자체가 없음 — 설치 후 취소돼도
          스타리온이 즉시 회수해가서 우리 재고로 안 들어옴
```

### 2-3. 마감 기준

- **양수(마감매입)**: **설치완료일** 기준 (접수일 아님). 이번달 주문이 다음달 설치되는 이월이 흔해서.
- **음수(취소차감)**: 취소상태=설치후취소 + 취소확정일 기준.

### 2-4. 설치완료 — `confirmStarionInstallation(saBangNo, installDate)`

```
조건: 완료여부=발주완료, 설치완료여부 공란, 취소상태≠설치전취소
처리: 설치완료여부=Y, 설치완료일 저장
화면: 표 안 인라인 버튼 (윈세이의 송장번호 입력처럼 평소 업무로 취급, 별도 탭 안 거침)
```

### 2-5. 취소 — `cancelStarionOrder(saBangNo, cancelDate, deductionAmount)`, 설치 전/후 자동 분기

```
설치완료여부 공란 (설치 전 취소)
  → 완료여부=주문취소, 취소상태=설치전취소, 매입 영향 없음

설치완료여부=Y (설치 후 취소)
  → 완료여부는 발주완료 유지, 취소상태=설치후취소
  → 취소확정일/취소차감금액 저장, 재고 복구 없음
  → 차감금액 검증: 0보다 커야 함, 원 주문금액 초과 불가
```

### 2-6. ⚠️ 헷갈리기 쉬운 부분 — "설치완료" 숫자가 두 군데서 다르게 보임

```
설치완료여부=Y는 한 번 찍히면 영구 이력 (설치 후 취소돼도 안 지움)

상단 "🏁 설치완료" 카드      = 설치완료여부=Y AND 취소상태≠설치후취소 (현재 유효한 것만)
"이번달 1차 매입마감" 설치완료건수/마감매입 = 설치완료여부=Y인 행 전부 포함 (이력 보존,
  취소되면 그 사실은 별도 음수 행으로 같은/다음 달에 반영됨)
```

즉 시트의 `설치완료여부` 컬럼 값과 화면 상단 카드 숫자가 일치하지 않을 수 있음 — 버그 아니라 의도된 설계.

### 2-7. 화면 구조

스타리온 서브뷰도 탭 2개: **📊 발주현황** / **❌ 취소처리** (윈세이는 "취소·반품"이지만 스타리온은 반품이 없어 "취소처리"로 이름 다름). 취소처리 탭은 카드 클릭 진입 없이 항상 주문번호 직접 조회.
