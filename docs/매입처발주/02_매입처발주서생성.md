## 1. 이 단계가 하는 일

구글시트 발주원장에서 매입처별 발주 대상 건을 읽어 각 매입처에 보낼 발주서(엑셀)를 생성한다.
3개 매입처를 하나의 프로그램이 순서대로 처리한다 (캐리어 → 윈세이 → 스타리온).

```
구글시트 캐리어_발주원장   (완료여부=신규)     ┐
구글시트 윈세이_발주원장   (완료여부=발주대기)  ├─→ 발주서생성.py 실행
구글시트 스타리온_발주원장 (완료여부=발주대기) ┘
        ↓
캐리어_발주서_YYYYMMDD.xlsx
윈세이_발주서_YYYYMMDD.xls
스타리온_발주서_YYYYMMDD.xlsx
        ↓
완료여부 자동 업데이트 (신규→발주중 / 발주대기→발주완료)
        ↓
(수동) 각 매입처에 이메일 발송
```

**매입처별 차이점 요약:**

| 항목 | 캐리어 | 윈세이 | 스타리온 |
|---|---|---|---|
| 대상 완료여부 | 신규 | 발주대기 | 발주대기 |
| 처리 후 완료여부 | 발주중 | 발주완료 | 발주완료 |
| 양식 | 풍림 표준 양식(xlsx) 템플릿 사용 | xlwt로 직접 생성(.xls) | openpyxl로 직접 생성(.xlsx) |
| 시트 구성 | 판매오더생성양식 1개 | 발주서 today / 발주서 total 2개 | TODAY / TOTAL 2개 |
| 단가 처리 | 이 단계에서 단가테이블 재조회 가능 (구글시트 값 없으면) | 구글시트 값 그대로 사용, 재조회 없음 | 구글시트 값 그대로 사용, 재조회 없음 |
| 날짜/비고/설치구분 자동계산 | 있음 (3~6번 항목) | 없음 | 없음 |
| 합계행 | 없음 | 있음 (수량 합계) | 없음 |

캐리어는 로직이 복잡해 별도 절을 충분히 두고(2~8번), 윈세이·스타리온은 9·10번에
간단한 구조만 정리한다 — 두 매입처는 구글시트 데이터를 거의 그대로 양식에 옮기기만 하기 때문이다.

## 6. 모델코드 특수 처리 — `-설치비제외` 접미사 ✅ 적용됨 (v2.4)

사방넷/발주총괄에서 모델코드 끝에 `-설치비제외`가 붙어서 들어오는 경우가 있다.
(예: `EARB-0162GDWSD-설치비제외`)

**처리 규칙:**
1. AE열(아이템)에는 접미사를 **제거한** 순수 모델코드만 넣는다. (`EARB-0162GDWSD`)
2. 단가는 단가테이블의 **공급가_설치제외** 컬럼 값을 사용한다. (단가테이블 자체가 이미 설치제외 단가이므로 그대로 조회)
3. 배송구분과 무관하게, 이 접미사가 붙은 건은 **배송상품(받는고객 Q~U열)** 으로 처리한다.
   - 설치비를 받지 않는 주문이므로 캐리어 설치 인력이 가지 않는다.
4. J열 설치비포함 코드는 `Z02`(설치비미포함).

## 7. 단가 결정 로직 ✅ 적용됨 (v2.4, 2026-06-19)

### 7-1. 배경 문제
캐리어 모델 중 `TARB-0062GDWSD`는 **홈쇼핑 전용 모델**이다.
운영안 파일에는 모델별로 `벤더 공통` 시트와 `홈쇼핑` 시트에 각각 단가가 있는데,
TARB는 홈쇼핑 시트 단가가 적용되어야 하지만 기존 코드는 채널명에 "홈"이라는 글자가
포함되어 있는지로 판단했다. 사방넷 점포명은 "Kt알파쇼핑", "신세계몰(신)" 등으로 들어오기 때문에
"홈"이라는 글자 매칭 자체가 무의미했고, 결과적으로 모든 캐리어 주문이 벤더공통 단가로
잘못 계산되고 있었다.

### 7-2. 적용된 임시 조치 (2026-06-19 ~ 직원 협의 전까지)

**단가 결정 순서 (우선순위):**
```
1순위: 구글시트 원장의 캐리어공급가_VAT제외 값 (1단계 매입처발주.py가 채워놓은 값)
2순위: 1순위가 비어있을 때만 — 이 단계(2단계)의 단가테이블 직접 조회
```

⚠️ 즉 **1단계에서 이미 단가를 채워서 구글시트에 넣어버리면, 2단계는 그 값을 그대로
믿고 쓴다.** 1단계 로직이 잘못되어 있으면 2단계 단가테이블을 아무리 정확히 고쳐도
소용없다 (구글시트 값이 항상 먼저 적용되므로). 실제로 2026-06-19 이 문제로 디버깅
시간이 걸렸다 — 1단계(01_매입처발주.md 4번 항목)도 함께 고쳐야 했다.

**프로그램은 채널을 구분하지 않고, 모든 캐리어 모델에 홈쇼핑 단가를 우선 적용한다.**
홈쇼핑 시트에 없는 모델은 자동으로 벤더공통 단가가 적용된다 (홈쇼핑 시트엔 일부 모델만 있음).

이유: 홈쇼핑 단가가 벤더공통 단가보다 낮으므로(저렴한 쪽), 일단 더 안전한(원가 손실이 적은)
방향으로 통일해서 발주서를 발송한다. 이후 직원들과 협의하여 모델별 채널 구분 코드를
사방넷에 추가하기로 결정되면 이 로직을 다시 세분화한다.

```python
# 변경 전 (틀린 로직 — 폐기, v2.3까지)
ch_key = "홈쇼핑" if "홈" in str(o.get("채널", "")) else "벤더공통"
supply_price = price_map.get((model, ch_key)) or price_map.get((model, "벤더공통"))

# 변경 후 (v2.4부터 적용 중)
supply_price = price_map.get((model, "홈쇼핑")) or price_map.get((model, "벤더공통"))
```

| 모델코드 | 홈쇼핑 공급가(설치제외) | 벤더공통 공급가(설치제외) |

### 7-3. 향후 정식 처리 (미결정 — 직원 협의 필요)
직원 협의 후, 모델별로 "이 모델은 채널 구분 없이 단일 단가"인지 "이 모델은 채널에 따라

## 8-6. 홈쇼핑 전용 모델 — AM열(할인율) "미리주문" 표시 ✅ 적용됨 (v2.5)

TARB-0062GDWSD, EARB-0102GDWSD로 발주서를 만들 때는 일반 모델과 다른 표시 규칙이 추가로 있다.

| 항목 | 일반 모델 | 홈쇼핑 전용 모델(TARB, EARB-0102) |
|---|---|---|
| AC/AD열 비고 | `빠른 출고`/`최근 생산` + 물류메시지 | `빠른 출고`/`최근 생산`만 (물류메시지 제외) |
| AM열(할인율) | 공란 | `미리주문` |

```python
HOMESHOPPING_MODELS_FOR_NOTE = ['TARB-0062GDWSD', 'EARB-0102GDWSD']
is_homeshopping_model = model in HOMESHOPPING_MODELS_FOR_NOTE
note_for_remark = "" if is_homeshopping_model else note   # 비고에서 물류메시지 제외
am_note = "미리주문" if is_homeshopping_model else None     # AM열 표시
```

이 목록(`HOMESHOPPING_MODELS_FOR_NOTE`)은 7번 항목의 단가 우선순위 판단에 쓰는
`HOMESHOPPING_MODELS`와 **모델 구성은 같지만 변수를 분리**해 두었다 — 단가 결정과
표시 규칙은 서로 다른 목적이라 나중에 모델 목록이 갈라질 가능성을 대비한 것.

> ✅ 상태: 2026-06-19 발주서생성.py v2.5에 반영. 6/19 발주 TARB 2건으로 검증, 정답파일과 일치 확인.
