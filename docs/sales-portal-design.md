# 풍림전자 매출관리포털 설계서

작성일: 2026-06-17  
문서 목적: 사방넷 주문 수집부터 오성물류 발주, 송장 수신, 사방넷 송장 업로드, 거래처 회신, ERP 업로드, 마감대조까지의 업무 흐름을 하나의 매출관리원장 중심으로 통합하기 위한 기준 문서.

---

## 1. 전체 시스템 방향

기존 업무는 사방넷에서 엑셀을 다운로드하고, 파이썬 프로그램을 실행해 여러 업로드 파일을 만들고, 다시 외부 시스템에 업로드하는 파일 중심 구조였다.

개편 방향은 다음과 같다.

```text
사방넷 = 주문 수집 / 판매채널 / 송장 업로드
풍림전자 매출관리포털 = 업무 통제센터 / 원장 / 상태관리 / 파일 생성 / AI 에이전트 작업공간
ERP = 출고 / 반품 / 교환 / 재고 / 회계 / 결산 확정 시스템
AI 에이전트 = 포털 데이터를 보고 검수·추천·파일생성·예외확인
```

핵심 원칙은 다음과 같다.

```text
파일은 임시 입출력물이다.
매출관리원장이 업무의 중심이다.
상태값은 업무 흐름이다.
버튼은 상태 전환 트리거다.
외부 시스템 반영 완료는 사람이 확인한다.
```

## 2. 구글시트 구조

구글시트 이름:

```text
풍림전자_매출관리원장
```

탭 구조:

```text
01_사방넷원본수집
02_주문상세원장
03_창고업로드이력
04_송장수신이력
05_사방넷송장업로드이력
06_거래처회신이력
07_ERP업로드이력
08_마감대조원장
90_실행로그
91_오류로그
99_상태값
```

각 탭의 역할:

| 탭 | 역할 |
|---|---|
| 01_사방넷원본수집 | 사방넷 다운로드 원본 보관. 사람이 수정하지 않음 |
| 02_주문상세원장 | 포털의 핵심 원장. 1행 = 사방넷주문번호 1개 |
| 03_창고업로드이력 | 오성 발주 파일 생성 및 업로드 완료 처리 이력 |
| 04_송장수신이력 | 오성 송장자료 업로드 및 매칭 이력 |
| 05_사방넷송장업로드이력 | 사방넷 송장 업로드 파일 생성 및 완료 처리 이력 |
| 06_거래처회신이력 | 거래처별 송장파일 생성 및 메일 발송 완료 처리 이력 |
| 07_ERP업로드이력 | ERP 업로드 파일 생성 및 완료 처리 이력 |
| 08_마감대조원장 | 마감대조 실행 결과 누적 기록 |
| 90_실행로그 | 버튼 실행, 파일 생성, 업로드, 상태 변경 로그 |
| 91_오류로그 | 사람이 확인해야 하는 오류 누적 |
| 99_상태값 | 상태값 및 코드표 |

## 3. 주문상세원장 기준

`02_주문상세원장`의 1행 기준은 다음으로 확정한다.

```text
1행 = 사방넷주문번호 1개
```

중요 기준:

```text
사방넷주문번호 = 실제 처리 단위
주문번호 = 고객 주문 묶음 번호
상품코드 = 사방넷 원본 상품 식별값
모델코드 = 내부 표준 모델명
물류처 = 창고/출고 분기 기준
점포명 = 거래처 송장 회신 기준
ERP업로드상태 = ERP 반영 기준
```

절대 `주문번호`를 1행 기준으로 삼지 않는다. 주문번호는 묶음 주문 그룹값이므로 복수 상품, 0원 상품, 동일 송장 상품이 섞일 수 있다.

## 7. 예외 규칙

### 예외 1. 비오성 물류처 직행 흐름

원장 최초 적재 시:

```text
물류처 ≠ 오성물류
→ 창고업로드상태 = 제외
```

비오성 주문은 아래 단계를 건너뛴다.

```text
1. 오성 업로드 파일 생성
2. 오성 업로드 완료 처리
3. 오성 송장자료 업로드
```

송장번호가 있으면 바로 후속 단계로 간다.

```text
송장수신상태 = 수신완료
사방넷송장업로드상태 = 대기
ERP업로드상태 = 대기 또는 대기전
```

### 예외 2. 오성 발주 후 고객 취소 제동 장치

조건:

```text
물류처 = 오성물류
AND 창고업로드상태 IN (파일생성완료, 업로드완료)
AND 현재주문상태 IN (취소접수, 취소완료, 반품접수, 반품회수준비, 반품완료, 교환접수 등)
```

자동 전환:

```text
오류상태 = 확인필요
오류유형 = 출고전취소
제동여부 = Y
제동사유 = 오성 발주 이후 취소/반품 계열 상태 감지
```

이 상태에서는 자동 후속 처리를 막는다.

```text
사방넷 송장업로드 파일 생성 제외
거래처별 송장파일 생성 제외
ERP 업로드 파일 생성 제외
마감 정상 처리 제외
```

사람이 오성 창고 출고 중단 여부를 확인하고 제동해제를 해야 다음 단계로 갈 수 있다.

## 8. 버튼 흐름

### 1. 오성 업로드 파일 생성

대상:

```text
물류처 = 오성물류
AND 현재주문상태 IN (신규주문, 주문확인)
AND 창고업로드상태 = 대기
AND 오류상태 = 정상
```

생성 파일:

```text
오성_발주서YYMMDD.xlsx
```

상태 변경:

```text
창고업로드상태 = 파일생성완료
창고업로드파일명 기록
창고업로드생성일시 기록
```

### 2. 오성 업로드 완료 처리

사람이 오성 창고 전산에 실제 업로드한 뒤 누르는 버튼.

상태 변경:

```text
창고업로드상태 = 업로드완료
창고업로드완료일시 기록
송장수신상태 = 대기 유지
```

### 3. 오성 송장자료 업로드

매칭 기준:

```text
오성 송장자료.주문번호 = 주문상세원장.사방넷주문번호
```

성공 시:

```text
택배송장번호 입력
송장수신상태 = 수신완료
사방넷송장업로드상태 = 대기
거래처회신상태 = 대기
```

실패 시:

```text
송장수신상태 = 미매칭
오류상태 = 확인필요
오류유형 = 송장미매칭
```

### 4. 사방넷 송장업로드 파일 생성

조건:

```text
송장수신상태 = 수신완료
AND 사방넷송장업로드상태 = 대기
AND 택배송장번호 있음
```

출력 컬럼:

```text
날짜
사방넷주문번호
송장번호
```

상태 변경:

```text
사방넷송장업로드상태 = 파일생성완료
사방넷송장업로드파일명 기록
```

### 5. 사방넷 송장 업로드 완료 처리

사람이 사방넷에 실제 업로드한 뒤 누르는 버튼.

상태 변경:

```text
사방넷송장업로드상태 = 업로드완료
사방넷송장업로드완료일시 기록
ERP업로드상태 = 대기
```

### 6. 거래처별 송장파일 생성

중요: 기준은 매입처가 아니라 점포명이다.

대상:

```text
택배송장번호 있음
AND 사방넷송장업로드상태 = 업로드완료
AND 거래처회신상태 = 대기
AND 점포명 ∈ 거래처_히스토리.발주총괄 점포명
AND 거래처_히스토리.메일발송 = Y
```

그룹화:

```text
점포명
→ 거래처_히스토리에서 거래처명 / 이메일 매핑
→ 판매 거래처별 송장전송 파일 생성
```

출력 컬럼:

```text
수집일자
인수자
주소
연락처
상품명
수량
택배사
송장번호
```

상태 변경:

```text
거래처회신상태 = 파일생성완료
거래처회신파일명 기록
거래처회신생성일시 기록
```

### 7. 거래처 메일 발송 완료 처리

아직은 자동 메일 발송하지 않는다. 사람이 직접 메일을 보낸 뒤 완료 처리한다.

상태 변경:

```text
거래처회신상태 = 메일발송완료
거래처회신발송일시 기록
거래처회신발송자 기록
```

### 8. ERP 업로드 파일 생성

현재 임시 기준:

```text
ERP업로드상태 = 대기
AND 현재주문상태 IN (출고대기, 취소접수, 취소완료)
AND 사용여부 = 정상
AND 오류상태 = 정상
```

현재 ERP는 기존 주문등록 업로드 로직을 기준으로 두되, 향후 ERP 화면을 보면서 출고/반품/교환 중심으로 재설계할 가능성을 열어둔다.

상태 변경:

```text
ERP업로드상태 = 파일생성완료
ERP업로드파일명 기록
ERP업로드생성일시 기록
```

### 9. ERP 업로드 완료 처리

사람이 ERP에 실제 업로드한 뒤 누르는 버튼.

상태 변경:

```text
ERP업로드상태 = 업로드완료
ERP업로드완료일시 기록
ERP업로드처리자 기록
마감대조상태 = 대기
```

### 10. 마감대조 실행

목적:

```text
포털 원장과 ERP 데이터를 비교해서 누락, 중복, 수량 차이, 금액 차이, 송장번호 차이, 상태 차이를 찾는다.
```

기본 대조 키:

```text
사방넷주문번호
```

초기 대조 항목:

```text
1. 사방넷주문번호 누락
2. 사방넷주문번호 중복
3. 수량 불일치
4. 금액 불일치
5. 송장번호 불일치
```

불일치 사유:

```text
ERP누락
ERP단독자료
원장중복
ERP중복
수량불일치
금액불일치
송장번호불일치
주문상태불일치
출고전취소제동
```

정상 시:

```text
마감대조상태 = 정상
마감대조일시 기록
```

불일치 시:

```text
마감대조상태 = 불일치
오류상태 = 확인필요
오류유형 = 마감불일치
오류내용 기록
```

## 9. 오성 업로드 파일 양식 및 로직

오성 양식 컬럼 17개:

```text
업체코드
업체명
주문번호1
주문번호2
주문번호3
수화인
주소
연락처
휴대전화
배송메시지
상품바코드
상품명
상품코드
상품코드2
컬러
사이즈
출고수량
```

원장 매핑:

```text
업체코드       ← 빈칸
업체명         ← 풍림전자
주문번호1      ← 빈칸
주문번호2      ← 사방넷주문번호
주문번호3      ← 빈칸
수화인         ← 정제수화인
주소           ← 통합주소 또는 인수자주소1
연락처         ← 인수자핸드폰번호
휴대전화       ← 인수자일반번호, 없으면 인수자핸드폰번호
배송메시지     ← 빈칸
상품바코드     ← 빈칸
상품명         ← 오성_상품코드테이블의 상품명
상품코드       ← 오성_상품코드테이블의 공급처코드/바코드
상품코드2      ← 모델코드
컬러           ← 빈칸
사이즈         ← 빈칸
출고수량       ← 주문수량 또는 분리수량
```

중요 로직:

```text
물류처에 오성 포함
주문상태 = 신규주문 / 주문확인
상품코드테이블로 바코드 매핑
NS-3000 + 치마형 키트 합포장 처리
NS-3000 수량 2~3개는 1개씩 행 분리
주소 누락 시 파일 생성 중단
배송메시지는 빈칸
수화인 이름 정제
```

## 10. ERP 관련 보류사항

ERP는 오늘 완전 확정하지 않는다.

현재는 기존 `ERP 업로드 파일 생성` 로직을 임시 기준으로 두되, 나중에 ERP를 직접 보면서 아래를 결정한다.

```text
ERP에 주문부터 넣을 것인가?
ERP에는 출고부터 넣을 것인가?
주문 없이 출고 입력이 가능한가?
반품/교환은 어떤 메뉴로 처리하는가?
ERP 업로드 양식은 주문등록용인가, 출고등록용인가?
```

현재 방향성:

```text
사방넷과 포털에서 주문을 관리하고,
ERP는 출고 / 반품 / 교환 / 재고 / 결산 중심으로 쓰는 방향 검토
```
