# 풍림전자 매출관리포털 설계서

문서 버전: v1.1  
최종 수정일: 2026-06-18  
문서 목적: 사방넷 주문자료를 기준으로 오성물류 발주, 송장 수신, 사방넷 송장 업로드, 거래처 송장 회신, ERP 업로드, 마감대조까지 처리하는 풍림전자 매출관리포털의 기준 설계를 정의한다.

---

## 1. 전체 시스템 방향

풍림전자 매출관리포털은 파일 생성 프로그램이 아니라, 사방넷과 ERP 사이의 업무 통제센터다.

```text
사방넷 = 주문 수집 / 판매채널 / 송장 업로드
풍림전자 매출관리포털 = 업무 통제센터 / 원장 / 상태관리 / 파일 생성 / 예외 검수
ERP = 출고 / 반품 / 교환 / 재고 / 회계 / 결산 확정 시스템
AI 에이전트 = 원장과 마스터 데이터를 기준으로 검수 / 파일 생성 / 예외 확인 보조
```

핵심 원칙:

```text
파일은 임시 입출력물이다.
매출관리원장이 업무의 중심이다.
상태값은 업무 흐름이다.
버튼은 상태 전환 트리거다.
외부 시스템 반영 완료는 사람이 확인한다.
```

## 2. 구글시트 구조

운영 구글시트명:

```text
풍림전자_매출관리원장
```

탭 구조:

```text
01_사방넷원본수집
02_주문상세원장
03_창고업로드이력
04_송장수신이력
05_사방넷송장업로드이력
06_거래처회신이력
07_ERP업로드이력
08_마감대조원장
90_실행로그
91_오류로그
92_품목마스터
93_거래처마스터
99_상태값
```

탭 역할:

| 탭 | 역할 |
|---|---|
| 01_사방넷원본수집 | 사방넷 원본 업로드 보관. 사람이 직접 수정하지 않는 원본 영역 |
| 02_주문상세원장 | 메인 운영 원장. 1행 = 사방넷주문번호 1개 |
| 03_창고업로드이력 | 오성 업로드 파일 생성 및 업로드 완료 처리 이력 |
| 04_송장수신이력 | 오성 송장자료 업로드 및 매칭 이력 |
| 05_사방넷송장업로드이력 | 사방넷 송장업로드 파일 생성 및 업로드 완료 처리 이력 |
| 06_거래처회신이력 | 거래처별 송장파일 생성 및 메일 발송 완료 처리 이력 |
| 07_ERP업로드이력 | ERP 업로드 파일 생성 및 업로드 완료 처리 이력 |
| 08_마감대조원장 | 포털 원장과 ERP/사방넷 자료의 마감 대조 결과 |
| 90_실행로그 | 버튼 실행, 파일 생성, 업로드 처리 등 시스템 실행 로그 |
| 91_오류로그 | 사람이 확인해야 하는 오류 / 제동 / 불일치 로그 |
| 92_품목마스터 | 상품/모델/오성 바코드/합포장 기준 마스터 |
| 93_거래처마스터 | 점포명 기준 거래처명/이메일/메일발송 기준 마스터 |
| 99_상태값 | 각 상태값 목록과 설명 |

## 3. 주문상세원장 기준

`02_주문상세원장`의 1행 기준은 다음과 같이 확정한다.

```text
1행 = 사방넷주문번호 1개
```

중요 기준:

```text
사방넷주문번호 = 실제 업무 처리 라인 단위
주문번호 = 고객 주문 묶음 번호
상품코드 = 사방넷 원본 상품 식별값
모델코드 = 내부 표준 모델명
물류처 = 창고/출고 흐름 분기 기준
점포명 = 판매 거래처 송장 회신 기준
ERP업로드상태 = ERP 반영 관리 기준
```

절대 금지:

```text
주문번호를 1행 기준으로 삼지 말 것.
```

이유: 주문번호는 묶음 주문 그룹값이므로 복수 상품, 복수 사방넷주문번호, 0원 상품, 동일 송장 상품이 섞일 수 있다.

## 6. 92_품목마스터

### 역할

`92_품목마스터`는 오성 업로드 파일 생성, 내부 모델코드 표준화, 바코드 매핑, 합포장 판단의 기준 테이블이다.

기존 외부 파일 `오성_상품코드테이블.xlsx`의 역할을 구글시트 내부로 가져온다. AI 에이전트와 Apps Script는 앞으로 이 탭을 기준으로 품목을 매핑한다.

## 9. 버튼 흐름

### 1. 오성 업로드 파일 생성

대상:

```text
물류처 = 오성물류 또는 물류처에 '오성' 포함
AND 현재주문상태 IN (신규주문, 주문확인)
AND 창고업로드상태 = 대기
AND 오류상태 = 정상
AND 사용여부 = 정상
```

생성 파일:

```text
오성_발주서YYMMDD.xlsx
```

생성 후 상태:

```text
창고업로드상태 = 파일생성완료
창고업로드파일명 = 생성 파일명
창고업로드생성일시 = 현재시간
창고업로드차수 = 1차
```

### 2. 오성 업로드 완료 처리

사람이 오성 창고 전산에 실제 업로드한 뒤 누르는 확인 버튼이다.

상태 변경:

```text
창고업로드상태 = 업로드완료
창고업로드완료일시 = 현재시간
송장수신상태 = 대기 유지
```

### 3. 오성 송장자료 업로드

매칭 기준:

```text
오성 송장자료.주문번호 = 02_주문상세원장.사방넷주문번호
```

성공 시:

```text
택배송장번호 입력
송장수신상태 = 수신완료
사방넷송장업로드상태 = 대기
거래처회신상태 = 대기
```

실패 시:

```text
송장수신상태 = 미매칭
오류상태 = 확인필요
오류유형 = 송장미매칭
```

### 4. 사방넷 송장업로드 파일 생성

## 11. 예외 규칙

### 11-1. 비오성 직행 흐름

원장 최초 적재 시:

```text
물류처 ≠ 오성물류
→ 창고처리대상여부 = N
→ 창고처리방식 = 비오성직행
→ 창고업로드상태 = 제외
```

비오성 주문은 1~3번 오성 흐름을 건너뛴다.

```text
1. 오성 업로드 파일 생성 제외
2. 오성 업로드 완료 처리 제외
3. 오성 송장자료 업로드 제외
```

송장번호가 원본에 이미 있으면:

```text
송장수신상태 = 수신완료
사방넷송장업로드상태 = 대기
```

송장번호가 없으면:

```text
송장수신상태 = 대기
사방넷송장업로드상태 = 제외 또는 대기전
```

### 11-2. 오성 발주 후 고객 취소 제동

조건:

```text
물류처 = 오성물류 또는 물류처에 '오성' 포함
AND 창고업로드상태 IN (파일생성완료, 업로드완료)
AND 현재주문상태 IN (취소접수, 취소완료, 반품접수, 반품회수준비, 반품완료, 교환접수)
```

자동 전환:

```text
오류상태 = 확인필요
오류유형 = 출고전취소
제동여부 = Y
제동사유 = 오성 발주 이후 취소/반품/교환 계열 상태 감지
```

이 상태에서는 후속 자동처리를 막는다.

```text
사방넷 송장업로드 파일 생성 제외
거래처별 송장파일 생성 제외
ERP 업로드 파일 생성 제외
마감 정상 처리 제외
```

### 11-3. 오성 출고 전 취소 제동 해제 시나리오

#### A. 출고차단성공

오성 창고에 연락하여 실제 출고를 막은 경우.

상태 변경:

```text
사용여부 = 제외
제동여부 = N
제동처리결과 = 출고차단성공
제동처리일시 = 현재시간
제동처리자 = 현재 사용자
오류상태 = 처리완료
처리완료여부 = Y
송장수신상태 = 제외
사방넷송장업로드상태 = 제외
거래처회신상태 = 제외
ERP업로드상태 = 제외
마감대조상태 = 제외
처리메모 = 오성 출고 차단 성공으로 주문 종결
```

주의:

```text
창고업로드상태는 되돌리지 않는다.
파일생성완료 또는 업로드완료 이력은 그대로 유지한다.
```

#### B. 출고차단실패

이미 출고되었거나 출고 중단이 불가능한 경우.

상태 변경:

```text
제동여부 = N
제동처리결과 = 출고차단실패
제동처리일시 = 현재시간
제동처리자 = 현재 사용자
오류상태 = 확인필요 또는 처리완료
송장수신상태 = 대기
사방넷송장업로드상태 = 대기전 또는 대기
거래처회신상태 = 대기전 또는 대기
ERP업로드상태 = 대기전 또는 대기
처리메모 = 출고 차단 실패, 실제 출고 흐름으로 후속 처리
```

이 경우 물건이 나간 것이므로 송장/사방넷/ERP 흐름을 계속 탄다. 다만 취소/반품 정산은 별도 확인이 필요하다.

### 매핑

```text

업체명         ← 풍림전자
주문번호1      ← 빈칸

주문번호3      ← 빈칸
수화인         ← 정제수화인
주소           ← 통합주소 또는 인수자주소1+인수자주소2
연락처         ← 인수자핸드폰번호
휴대전화       ← 인수자일반번호, 없으면 인수자핸드폰번호

상품바코드     ← 92_품목마스터.오성바코드
상품명         ← 92_품목마스터.오성상품명
상품코드       ← 92_품목마스터.오성상품코드
상품코드2      ← 92_품목마스터.상품코드2 또는 모델코드
컬러           ← 빈칸
사이즈         ← 빈칸
출고수량       ← 주문수량 또는 분리수량
```

### 주요 로직

```text
물류처에 오성 포함
현재주문상태 = 신규주문 / 주문확인
92_품목마스터로 바코드/상품명/상품코드 매핑
주소 누락 시 파일 생성 중단
배송메시지는 빈칸
수화인 이름 정제
```

### NS-3000 수량 분리

```text
NS-3000 수량 1개: 1행
NS-3000 수량 2~3개: 1개씩 행 분리
NS-3000 수량 4개 이상: 한 행 유지
```

### NS-3000 + 치마형 키트 합포장

본품 NS-3000과 치마형 키트가 같은 주문번호 또는 같은 수취인/주소/연락처 묶음으로 들어오면 합포장 처리한다.

복합 규칙:

```text
NS-3000 수량 2~3개 분리와 치마형 키트 합포장이 동시에 발생하는 경우,
합포장 대상 부속품은 분리된 첫 번째 NS-3000 행에만 결합한다.
나머지 NS-3000 분리 행에는 부속품을 복사하지 않는다.
```

예시:

```text
주문:
NS-3000 3개
치마형 키트 1개

출력:
1행: NS-3000 + 치마형 키트 합포장 코드
2행: NS-3000 단독
3행: NS-3000 단독
```

절대 금지:

```text
1행: NS-3000 + 키트
2행: NS-3000 + 키트
3행: NS-3000 + 키트
```

## 14. ERP 업로드 로직 보류사항

현재는 기존 `erp업로드생성.py` 기준으로 임시 설계한다.

현재 임시 대상:

```text
현재주문상태 IN (출고대기, 취소접수, 취소완료)
```

그러나 최종 방향은 보류한다.

검토할 사항:

```text
ERP에 주문부터 넣을 것인가?
ERP에는 출고부터 넣을 것인가?
주문 없이 출고 입력이 가능한가?
반품/교환은 어떤 메뉴로 처리하는가?
현재 ERP 업로드 양식은 주문등록용인가 출고등록용인가?
```

현재 판단:

```text
사방넷과 포털에서 주문을 관리하고,
ERP는 출고 / 반품 / 교환 / 재고 / 결산 중심으로 쓰는 방향을 검토한다.
```
