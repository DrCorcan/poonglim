# 풍림전자 매출관리포털 AI 에이전트 인수인계 문서

문서 버전: v1.1  
최종 수정일: 2026-06-18  
목적: 새 AI 에이전트 또는 새 ChatGPT 대화창이 풍림전자 매출관리포털 프로젝트의 핵심 맥락을 빠르게 복원하기 위한 문서다.

---

## 1. 프로젝트 목적

풍림전자 매출관리포털은 사방넷 주문자료를 중심으로 다음 업무를 하나의 원장과 상태값으로 관리하는 시스템이다.

```text
사방넷 주문 수집
오성물류 업로드 파일 생성
오성 송장자료 수신
사방넷 송장 업로드 파일 생성
거래처별 송장 회신 파일 생성
ERP 업로드 파일 생성
마감대조
오류/제동 관리
```

이 프로젝트의 핵심은 파일 생성 프로그램이 아니라 `02_주문상세원장`을 중심으로 한 업무 통제 시스템이다.

---

## 2. 절대 기준

### 2-1. 주문상세원장 1행 기준

```text
02_주문상세원장 1행 = 사방넷주문번호 1개
```

절대 금지:

```text
주문번호를 1행 기준으로 삼지 말 것.
```

이유:

```text
주문번호는 묶음 주문 번호다.
사방넷주문번호가 실제 처리 라인 단위다.
```

---

### 2-2. 거래처 회신 기준

```text
거래처 송장 회신 기준은 매입처가 아니다.
점포명 기준이다.
```

매칭:

```text
02_주문상세원장.점포명
= 93_거래처마스터.발주총괄점포명
```

---

### 2-3. 품목/바코드 기준

```text
오성 업로드 파일의 상품명, 상품코드, 상품바코드, 상품코드2는 92_품목마스터를 기준으로 생성한다.
```

외부 엑셀 파일명을 추정하지 말고, 구글시트 내부의 `92_품목마스터`를 기준으로 읽어야 한다.

---

### 2-4. Upsert 보존 원칙

사방넷 원본을 다시 업로드할 때 기존 행을 통째로 덮어쓰면 안 된다.

```text
사방넷주문번호가 없으면 신규 행 생성
사방넷주문번호가 있으면 기존 행 업데이트
단, 포털 내부 운영 상태값과 송장번호, 오류/제동/처리메모는 절대 덮어쓰지 않는다.
```

절대 덮어쓰면 안 되는 대표 필드:

```text
창고업로드상태
창고업로드파일명
창고업로드완료일시
택배송장번호
송장수신상태
사방넷송장업로드상태
거래처회신상태
ERP업로드상태
마감대조상태
오류상태
오류유형
오류내용
제동여부
제동사유
처리메모
```

주문상태 변경 시:

```text
기존 현재주문상태 → 이전주문상태
신규 사방넷 주문상태 → 현재주문상태
상태 변경 시 주문상태변경일시 기록
```

## 3. 공식 구글시트 탭 구조

운영 시트명:

```text
풍림전자_매출관리원장
```

공식 탭:

```text
01_사방넷원본수집
02_주문상세원장
03_창고업로드이력
04_송장수신이력
05_사방넷송장업로드이력
06_거래처회신이력
07_ERP업로드이력
08_마감대조원장
90_실행로그
91_오류로그
92_품목마스터
93_거래처마스터
99_상태값
```

중요:

```text
92_품목마스터와 93_거래처마스터는 필수 탭이다.
이 탭들이 없으면 오성 매핑과 거래처 회신 로직이 불완전하다.
```

## 4. 버튼 흐름

```text
1. 오성 업로드 파일 생성
2. 오성 업로드 완료 처리
3. 오성 송장자료 업로드
4. 사방넷 송장업로드 파일 생성
5. 사방넷 송장 업로드 완료 처리
6. 거래처별 송장파일 생성
7. 거래처 메일 발송 완료 처리
8. ERP 업로드 파일 생성
9. ERP 업로드 완료 처리
10. 마감대조 실행
```

외부 시스템에 실제 업로드했는지 여부는 사람이 확인한다.

```text
파일생성완료 ≠ 업로드완료
```

## 5. 비오성 직행 흐름

비오성 물류처는 오성 창고업로드 흐름을 타지 않는다.

원장 최초 적재 시:

```text
물류처 ≠ 오성물류
→ 창고처리대상여부 = N
→ 창고처리방식 = 비오성직행
→ 창고업로드상태 = 제외
```

비오성 주문은 아래 버튼을 건너뛴다.

```text
1. 오성 업로드 파일 생성
2. 오성 업로드 완료 처리
3. 오성 송장자료 업로드
```

## 6. 오성 출고 전 취소 제동

조건:

```text
물류처 = 오성물류 또는 물류처에 '오성' 포함
AND 창고업로드상태 IN (파일생성완료, 업로드완료)
AND 현재주문상태 IN (취소접수, 취소완료, 반품접수, 반품회수준비, 반품완료, 교환접수)
```

자동 전환:

```text
오류상태 = 확인필요
오류유형 = 출고전취소
제동여부 = Y
제동사유 = 오성 발주 이후 취소/반품/교환 계열 상태 감지
```

`제동여부 = Y`인 주문은 후속 파일 생성 대상에서 제외한다.

---

## 7. 오성 출고 전 취소 제동 해제 시나리오

### 7-1. 출고차단성공

오성 창고에 연락하여 실제 출고를 막은 경우.

상태 처리:

```text
사용여부 = 제외
제동여부 = N
제동처리결과 = 출고차단성공
오류상태 = 처리완료
처리완료여부 = Y
송장수신상태 = 제외
사방넷송장업로드상태 = 제외
거래처회신상태 = 제외
ERP업로드상태 = 제외
마감대조상태 = 제외
```

주의:

```text
창고업로드상태는 되돌리지 않는다.
파일생성완료 또는 업로드완료 이력은 유지한다.
```

### 7-2. 출고차단실패

이미 출고되었거나 출고 중단이 불가능한 경우.

상태 처리:

```text
제동여부 = N
제동처리결과 = 출고차단실패
송장/사방넷/거래처/ERP 흐름을 계속 진행
처리메모에 출고차단실패 기록
```

## 8. NS-3000 분리 + 합포장 규칙

### 수량 분리

```text
NS-3000 수량 1개: 1행
NS-3000 수량 2~3개: 1개씩 행 분리
NS-3000 수량 4개 이상: 한 행 유지
```

### 치마형 키트 합포장

본품 NS-3000과 치마형 키트가 같은 주문번호 또는 같은 수취인/주소/연락처 묶음이면 합포장 처리한다.

복합 규칙:

```text
NS-3000 수량 2~3개 분리와 치마형 키트 합포장이 동시에 발생하는 경우,
합포장 대상 부속품은 분리된 첫 번째 NS-3000 행에만 결합한다.
나머지 NS-3000 분리 행에는 부속품을 복사하지 않는다.
```

예시:

```text
주문:
NS-3000 3개
치마형 키트 1개

출력:
1행: NS-3000 + 치마형 키트 합포장 코드
2행: NS-3000 단독
3행: NS-3000 단독
```

절대 금지:

```text
분리된 모든 NS-3000 행에 치마형 키트를 복사하지 말 것.
```

## 9. ERP 관련 현재 상태

ERP 로직은 아직 최종 확정이 아니다.

현재는 기존 `erp업로드생성.py` 기준으로 임시 설계한다.

임시 대상:

```text
현재주문상태 IN (출고대기, 취소접수, 취소완료)
```

보류사항:

```text
ERP에 주문부터 넣을 것인가?
ERP에는 출고부터 넣을 것인가?
주문 없이 출고 입력이 가능한가?
반품/교환은 어떤 메뉴로 처리하는가?
현재 ERP 업로드 양식은 주문등록용인가 출고등록용인가?
```

현재 방향:

```text
사방넷과 포털에서 주문을 관리하고,
ERP는 출고 / 반품 / 교환 / 재고 / 결산 중심으로 쓰는 방향을 검토한다.
```

AI 에이전트는 ERP 로직을 임의로 확정하지 말 것.

## 12. 새 AI 에이전트가 먼저 해야 할 일

새 대화창 또는 새 AI 에이전트는 다음 순서로 맥락을 복원한다.

```text
1. 이 문서 sales-portal-agent-guide.md를 먼저 읽는다.
2. 상세 설계가 필요하면 sales-portal-design.md를 읽는다.
3. 변경 내역은 sales-portal-changelog.md를 확인한다.
4. 사용자가 제공한 최신 구글시트/Apps Script 상태를 확인한다.
5. 임의로 주문번호 기준, 매입처 기준, 전체 덮어쓰기 로직을 만들지 않는다.
```

절대 임의 변경 금지 항목:

```text
02_주문상세원장 1행 기준 = 사방넷주문번호
거래처 회신 기준 = 점포명
품목 매핑 기준 = 92_품목마스터
거래처 매핑 기준 = 93_거래처마스터
사방넷 원본 Upsert 시 포털 내부 상태값 보존
오성 취소 제동 시 후속 처리 중단
NS-3000 합포장 부속품은 첫 번째 분리 행에만 결합
ERP 로직은 아직 임시
```
